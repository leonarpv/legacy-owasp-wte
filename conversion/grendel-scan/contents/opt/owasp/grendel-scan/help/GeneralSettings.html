Internal Proxy Settings

Grendel-Scan supports a internal proxy server that can be used to guide the scan through an application. This is particularly useful for complicated applications, such as ones using AJAX or JSON. As you browse the application through this proxy server, Grendel-Scan will test the application components that you request. This is a safer alternative to the spider modules.

Proxy bind address

This is the IP address that the proxy will bind to. To bind to all available IP addresses, use "0.0.0.0". Generally, the default of localhost (127.0.0.1) should be used. 

Proxy bind port

This is the port that the proxy will listen on. For some operating systems, you will need to run with root privileges to bind to any port <= 1024. 

Max proxy threads

These threads are only used for accepting and processing proxy requests. They are not used for any testing. Under most circumstances, two threads should be plenty.

Thread Settings

Max requester threads

The requester threads service the requester queue, which is primarily used for the spider. If you are not using any of the spider modules, set this to "1", but never to "0".

Max tester threads

The tester threads service the testing queue, which is where most of the heavy lifting is done. Most HTTP requests initiated by testing modules are requested using these threads, not the requester threads (which are used for spidering). If the performance bottleneck does not appear to be the network, and you have CPU cycles to spare, increasing the number of tester threads may improve performance. If the thread count is significantly above the default, total memory available should also be increased. This can be done in the launch script.

Max categorizer threads

The categorizer threads service the categorizer queue. When an HTTP transaction is completed, it is sent to the categorizer queue to determine which tests should be run against it. The default value should be sufficient, unless the categorizer queue is consistently above zero.

Output Settings

Output directory

The output directory is the default location for any data saved to disk. This includes the report file, the HTTP transaction record, the error log, and output from modules (such as the website mirror).

Base URLs

The base URLs are primarily used to seed the spider. It also adds an implicit URL white-list rule based on the scheme, hostname, and port number of the provided URL.

Reporting

Report type

Currently, only text and HTML formated reports are available. The text format still has some issues, so HTML is the best bet. PDF and XML formated reports are on the roadmap.

Filename

The output filename for the report. Make sure that it's a valid filename, path, permissions, etc.

Minimum severity

The minimum severity of event to include in the report. Usually, "informational" is the best value.
